## Key Technical Decisions

### 1. Role-Based Navigation
- **Decision**: Use provider to detect role and show different scaffolds
- **Rationale**: Clean separation, easy to extend with more roles
- **Alternative**: Could use route guards, but scaffold switching is simpler

### 2. Shared Services Provider
- **Decision**: Same provider for both client and barber views
- **Rationale**: Ensures data consistency, real-time updates
- **Implementation**: Made autoDispose to allow refresh after CRUD

### 3. Two-Tab Admin Design
- **Decision**: Schedule and Services in one screen with tabs
- **Rationale**: Common admin dashboard pattern, efficient navigation
- **Alternative**: Could be separate screens, but tabs are more compact

### 4. Floating Action Button for Create
- **Decision**: FAB for adding services
- **Rationale**: Material Design standard, discoverable
- **Alternative**: Could use app bar button, but FAB is more prominent

### 5. Modal Dialogs for CRUD
- **Decision**: Dialog for create/edit/delete forms
- **Rationale**: Keeps user context, quick interactions
- **Alternative**: Could use full-screen forms, but dialogs are faster

## Security Considerations

1. **Database-Level Security**: User role stored in database, not just client
2. **RLS Policies**: Should implement Row Level Security in Supabase
3. **API Validation**: All operations validated on server side
4. **Session Management**: Uses Supabase auth for secure sessions

## Performance Optimizations

1. **AutoDispose Providers**: Automatically clean up when not needed
2. **Lazy Loading**: Appointments and services load only when tab is viewed
3. **Optimistic Updates**: UI updates before server confirmation (where safe)
4. **Minimal Rebuilds**: Consumer widgets only rebuild when their data changes
